01 · Thesis Decomposition

왜 뉴스 트레이딩은 여전히 알파가 남아있는가

방향성 판단 가능 + 집단 반응 = 변동성
Edge 1 · 속도 (Speed Edge)
엄청난 에너지 투입 → 정보→해석→대응을 초·분 단위로 압축
  • • 24/7 파이프라인, 중복 없는 스트리밍 소스
  • • 분류기가 거래 가능(actionable)한 이벤트만 통과
  • • 주문기는 멱등 (client_order_id 기반 중복 방지)
  • • 레이턴시 예산: 공지→주문 < 5초, KOL→주문 < 30초
Edge 2 · 해석 (Interpretation Edge)
진입이 늦어도 구조적 영향을 먼저 읽으면 알파
  • • 이벤트가 일시적 노이즈인지 서사 파괴인지 구분
  • • 2차 전염 경로 (담보·LP·브리지·서브넷 연결) 매핑
  • • 유사 과거 사례 검색 (XPL Hyperliquid, LUNA, FTX …)
  • • LLM = 분류 보조, 포지션 사이징은 규칙 기반
0xSun이 말하지 않은 진실
Survivorship bias 경고 · 복기는 오른쪽 꼬리만 보여준다
  • • 6사례는 사후 선택된 winner들
  • • 동기간 실패한 "이벤트 페이드" 거래가 훨씬 많을 것
  • • Arc/Rave 숏 진입: 락커·마켓메이커 역저격 리스크 실존

signal_hub
모든 이벤트가 지나는 단일 토픽 버스
DDD Core
무엇
소스(KOL·공지·거래량·지갑·DeFi)와 소비자(분류기·전략·실행기) 사이의 단일 메시지 배관. 소스 N개 × 소비자 M개의 직결을 N+M 연결로 축소해, 한쪽을 교체해도 나머지를 안 건드린다.
어떻게
Redis Streams 3단 토픽: signals.raw → signals.classified → trades.planned. Producer는 XADD, Consumer는 XREADGROUP + XACK로 at-least-once. 각 이벤트 ID는 sha256(source|url|ts)로 전역 유일 → 재전송돼도 무해.
입력 → 출력
IN: 모듈들이 발행하는 Python dataclass 메시지
OUT: 각 스테이지에 구독 중인 소비자로 브로드캐스트
왜
전 사례 공통 배관. 이게 없으면 "KOL 트윗 → 주문기 직접 호출"처럼 로직이 얽히고, 하나 고장 시 전체 연쇄 실패.
함정
메시지 보존 정책 (MAXLEN ~ 10K), Dead-letter queue, 재시작 시 consumer offset 유실 방지.

무엇

심볼별 20일 1분봉 거래량의 rolling μ, σ를 유지. 새 분봉이 닫히면 Z = (vol − μ)/σ 계산 · 동시에 진폭 = (high−low)/mid. Z > 6 AND 진폭 < 1%면 이상 감지 → 중심가 기준 ±0.3/0.5/0.7%에 Ladder 지정가. Paper mode 2주 검증 후에만 실전.

OUT: RawSignal{kind:'vol_anomaly', symbol, center, width, levels}

08 · Reality Check

0xSun이 말하지 않은 6가지 리스크

Risk 1
Survivorship

복기는 이긴 거래만 보여준다. 동시기간 실패한 이벤트 페이드를 저널에 강제 기록.

Risk 2
페이크 뉴스

조작된 트윗·딥페이크·가짜 공지 가능. KOL + 온체인 2소스 교차 확인 전 진입 금지.

Risk 3
유동성 함정

숏 진입 시점이 바닥. Ladder + max position, 단일 틱 체결 한도 설정.

Risk 4
역저격

지갑 클러스터 추적은 세력도 본다. 공개 추적이 숏스퀴즈 재료가 되는 역사 多.

Risk 5
레이턴시 경쟁

0.1초 차이로 가격이 이미 반영된 경우 多. 1분 지나면 얼리 알파 소멸.

Risk 6
LLM slop

분류기가 가짜 심볼·허상 이벤트 생성 가능. 실제 CEX/DeFi에 존재하지 않는 심볼은 거부.
